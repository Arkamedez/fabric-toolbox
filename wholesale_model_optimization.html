3. TODAY() / NOW() in 34 Measures — Prevents Query Caching

[!CAUTION]
TODAY() is a volatile function. In DirectQuery mode, it prevents Snowflake query result caching because the query parameters change every day. With 34 measures using TODAY(), this means these queries can never hit the Snowflake result cache.

Key offenders:

  • Total Sales Amount, Shipped Case Qty, Not Shipped (Buyer/Supply Chain/Unauth/Vendor) — all use FILTER('D_Time Hierarchy', Full Date <= TODAY())
  • Multiple < today report measures
  • F_PO Compliance has TODAY() as a calculated column (entire table re-evaluated daily)

Recommendation: Replace TODAY() references with a pre-computed "current date" column or a disconnected date parameter table that's set at refresh time. Example:

// Instead of:
FILTER('D_Time Hierarchy', 'D_Time Hierarchy'[Full Date] <= TODAY())

// Use a pre-computed flag column (import or Snowflake view):
FILTER('D_Time Hierarchy', 'D_Time Hierarchy'[Is Current Or Past] = TRUE)

🟠Important Issues (Medium Impact)

4. 175 Measures Using FILTER() — Many with Expensive Patterns

95+ measures use FILTER(ALL('D_Time Hierarchy'), ...) — the classic "Last Year" pattern:

CALCULATE(
    SUM('F_FD.Sales'[Total Sales Amt]),
    FILTER(
        ALL('D_Time Hierarchy'),
        'D_Time Hierarchy'[Date Key] IN VALUES('D_Time Hierarchy'[Date ID LY])
    )
)
[!IMPORTANT]
FILTER(ALL('D_Time Hierarchy'), ...) iterates the entire date table to find matching rows. In DirectQuery, this generates a subquery or cross-join that Snowflake must evaluate. With 95+ measures doing this, query complexity compounds massively.

Recommendations:

  • Use TREATAS() or USERELATIONSHIP() for LY calculations instead of FILTER(ALL(...))
  • The Time Intelligence calculation group (PTD/QTD/YTD/WTD) already exists — extend it to handle LY comparisons natively, reducing measure count
  • Consider using a DATEADD-style approach via the calculation group rather than individual LY measures

🟡Optimization Opportunities (Lower Impact but Worth Pursuing)

7. Massive Measure Count (1,072) — Many Appear Duplicative

The model has two giant measure tables:

  • O_Ad-hoc Measures: 296 measures
  • O_Report Measure: 763 measures

There are apparent duplicates with different naming conventions:

  • LY Total Sales Amount (ad-hoc) vs LY Total Sales Amt (report)
  • Shipped Qty / Shipped Case Qty / LY Shipped Qty duplicated across both tables
  • Count Orders, Count_Orders, Count-Orders, Count Orders. — 4 variants of the same measure!

Recommendation: Consolidate measures. This reduces DAX compilation time and improves maintainability. Aim to get under 500 measures.

8. F_Agg_Sales Aggregation Table — Validate Coverage

The model has an aggregation table F_Agg_Sales (import, incremental refresh, 13 partitions) mapping 27 columns from F_FD.Sales. This is good practice for DirectQuery optimization.

Verify:

  • Are the aggregation mappings covering the most common query patterns (i.e., the sales dashboard KPIs)?
  • The agg table uses 10-year rolling window with 2-month incremental — this seems appropriate
  • Ensure the Summarization properties are correctly set for all GroupBy vs. Sum columns
  • Consider adding more aggregation tables for other heavy DQ fact tables like F_Purchase Detail or F_PO Payment Detail

9. D_Item Has 14 Calculated Columns in Dual Mode

The D_Item table (272 columns, dual mode) has 14 calculated columns. In dual mode, these are computed during import refresh but still affect import refresh time and model size.

Recommendation: Move these to the Snowflake view to eliminate DAX-computed columns entirely.
